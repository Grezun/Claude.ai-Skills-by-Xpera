# Deterministic PDF Parser

## Overview

**Never read a PDF with the Read tool and answer from that output alone.**

The Read tool extracts PDF text as a flat character stream — multi-column layouts, tables, and financial statements all come back scrambled. Answering from that output means answering from corrupted data.

Instead, always run `parse_pdf.py` first. Feed its raw markdown output to yourself, then answer. This is the only safe path.

## Core Pattern

```
1. Run parse_pdf.py on the PDF via Bash tool
2. The script prints markdown to stdout — read it from the Bash tool result directly
3. Answer from that output — not from assumptions, not from Read tool
```

**The output comes from stdout, not a file.** Run via Bash and read the result inline:

```bash
# Full document
python3 /path/to/skills/deterministic-pdf-parser/scripts/parse_pdf.py /path/to/file.pdf

# Specific pages
python3 /path/to/skills/deterministic-pdf-parser/scripts/parse_pdf.py /path/to/file.pdf --pages 3
python3 /path/to/skills/deterministic-pdf-parser/scripts/parse_pdf.py /path/to/file.pdf --pages 2-5
```

**Install if needed (do this, don't skip the script):**
```bash
pip install pdfplumber pypdf
```

**If the PDF is password-protected**, the script will error. Tell the user to provide the password or an unlocked copy — do not attempt to extract without it.

## When to Use

Use for any PDF extraction request:
- "Extract all transactions from this statement"
- "What are the total charges on my bill?"
- "Get the table from page 3"
- "Summarise this report"
- "Read this invoice"

**Always run the script.** Don't decide first whether the task is "simple enough" to skip it.

## When NOT to Use

- The file is an image (`.png`, `.jpg`) not a PDF — show the image directly instead
- You have already run the script this session and the output is still in context — use that output, don't re-run

## Quick Reference

| Task | Command |
|---|---|
| Full document | `python3 parse_pdf.py file.pdf` |
| Single page | `python3 parse_pdf.py file.pdf --pages 3` |
| Page range | `python3 parse_pdf.py file.pdf --pages 1-5` |
| Tables auto-detected | Output includes `### Table N` sections |
| Scanned page detected | Script prints `⚠ SCANNED IMAGE PAGE` |

## Scanned PDFs

If the script prints `⚠ SCANNED IMAGE PAGE` for any page, those pages contain images rather than selectable text. Tell the user:

> "Page X appears to be a scanned image — I can't extract text from it without OCR. Options: Adobe Acrobat's OCR tool, Google Drive (auto-OCRs on upload), or share a screenshot of the page and I can read it directly."

Do not guess at the content of scanned pages.

## Common Mistakes

### ❌ Reading the PDF with the Read tool and answering from it

```
# WRONG — Read returns flat character stream, tables are scrambled
Read("/tmp/statement.pdf")
→ "The total is $142.50"  ← may be wrong, unverifiable
```

The Read tool works for plaintext files. PDF is not plaintext.

### ❌ Skipping the script because the task "seems simple"

> "I just need one number — overkill to run a script."

There is no "simple" PDF extraction. The corrupted data risk is the same whether you want one number or a full table. Run the script.

### ❌ Using pdftotext -layout as a substitute

`pdftotext -layout` preserves character spacing but does not detect table structure geometrically. It frequently merges columns. `parse_pdf.py` uses `pdfplumber`'s bounding-box analysis — these are not equivalent.

### ❌ Assuming "machine-generated = clean"

Machine-generated PDFs (invoices, statements) still use multi-column layouts and floating elements that break text stream order. "Generated by software" does not mean "safe to read as plain text."

### ❌ Presenting a number before running the script

Never write a dollar amount, date, or figure before the script has run. If the user is impatient, say:

> "Running the extractor now — won't take a moment."

Then run it.

## Rationalization Table

| Excuse | Reality |
|---|---|
| "The Read tool handles PDFs natively" | It returns a character stream. Tables, columns, and footers all merge. |
| "I'll use Python only if Read fails" | Read always "succeeds" — it returns something. That something may be wrong. Run Python first. |
| "It's over-engineering for a simple task" | There are no simple PDF tasks. The extraction risk is identical. |
| "pdftotext -layout is good enough for tables" | It is not. It has no geometric awareness. Use pdfplumber. |
| "This is a machine-generated PDF, it'll be clean" | Multi-column machine-generated PDFs break Read tool output just as badly. |
| "I already know what it says" | You don't. Run the script. |
| "pdfplumber isn't installed, I'll skip it" | Install it. `pip install pdfplumber pypdf` takes seconds. Don't skip the script because a dependency is missing. |

## Red Flags — STOP

- You are about to call Read on a `.pdf` file
- You are about to write a number from a PDF before running the script
- You are thinking "this PDF is probably clean"
- You are thinking "just running grep/pdftotext will be fine here"
- You have already answered and realize you didn't run the script
- You ran the script but are now trying to Read a separate output file (the output is in the Bash result — use that)

All of these mean: **run `parse_pdf.py` before proceeding.**
